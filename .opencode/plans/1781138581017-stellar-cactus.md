## Part 2: Can We Use OCCT Directly?

**Short answer:** Yes, but not through the current `occt-wasm` package. That WASM binary excludes the TKXCAF/TKDESTEP modules needed for PMI.

OCCT's PMI classes live in two toolkits:
- **TKXCAF** — `XCAFDoc_DimTolTool`, `XCAFDimTolObjects_GeomToleranceObject`, `XCAFDoc_Dimension`, etc. (label-based GD&T document model)
- **TKDESTEP** — `StepDimTol_GeometricTolerance`, `STEPCAFControl_Reader` (STEP I/O with PMI persistence)

These modules are compiled into OCCT's C++ library but the current occt-wasm facade simply doesn't expose them. The `andymai/occt-wasm` project omits them intentionally — it's assembly/XCAF-lite only.

### Integration Options (ranked by pragmatism)

---

#### Option A: Sidecar Native Process ⭐ **Recommended**

```
MCP Server (TypeScript)  ──stdin/stdout JSON──>  occt-sidecar (C++/OCCT)
  - Fully portable           Request:                - Reads STEP
  - Pure TypeScript          {"method":"pmi",        - XCAFDoc_DimTolTool
  - No WASM overhead           "params":             - Returns JSON
  - No native modules           {file:"..."}}
```

**How it works:**
- A small C++ program (~200 lines) links against OCCT's TKXCAF + TKDESTEP
- Reads JSON-RPC requests from stdin, writes responses to stdout
- MCP server `spawn()`s it once, pipes JSON back and forth
- On shutdown, `kill()` the child process

**Key advantage:** The MCP server stays **pure TypeScript** — zero native compilation, zero WASM bundles. The sidecar binary is the only compiled artifact and can be distributed per-platform.

**Distribution model (proven by `occt-gltf-addon`):**
```
cad-mcp-server (pure TS npm package)
  └── optionalDependencies:
        cad-mcp-sidecar-linux-x64   (5MB)
        cad-mcp-sidecar-darwin-arm64
        cad-mcp-sidecar-win32-x64
```

**Pros:**
- Full OCCT API access (geometry + PMI + anything)
- Near-native performance (no WASM interpretation)
- MCP server stays portable
- Independent deployment (sidecar can be updated separately)
- Simple architecture (stdin/stdout JSON = MCP itself)

**Cons:**
- Need to build per-platform binaries
- Subprocess management (crashes, timeouts)
- Serialization overhead (but PMI queries are small payloads)

---

#### Option B: N-API Native Addon

Same as sidecar but loaded into the Node.js process as a `.node` file (like `occt-gltf-addon` does).

**Pros:** No subprocess overhead. Direct function calls from JS to C++.
**Cons:** Ties Node.js version to addon ABI. Harder to debug. Can crash the host process.

Sidecar is strictly better for an MCP server — isolation, independent lifecycle, simpler distribution.

---

#### Option C: Full OCCT WASM (OpenCascade.js custom build)

Use `opencascade.js@beta`'s Docker-based build system to create a WASM with only needed TKXCAF/TKDESTEP symbols.

**Pros:** Fully portable (no native binaries). No subprocess.
**Cons:** Project has 75 open issues, last push Aug 2023, possibly dormant. Slow init (3-5s). 3-4GB RAM. Raw OCCT JS API is verbose. Requires Docker in build pipeline.

---

#### Option D: Fork occt-wasm + Add DimTol Facade

Add `XCAFDoc_DimTolTool` methods to the `andymai/occt-wasm` facade C++ code and rebuild WASM.

**Pros:** Leverages existing facade pattern. WASM portable.
**Cons:** Need to maintain a fork. Rust/C++ build pipeline. Author explicitly limits scope. High effort.

---

### Recommendation

| Layer | Now | Future |
|---|---|---|
| **Geometry queries** (faces, edges, B-rep, compare) | Keep `occt-wasm` (it works, fast, proven) | Keep `occt-wasm` |
| **PMI/GD&T** | Fix `surfaceNormal` bug + improve lightweight parser | Build **sidecar** for full OCCT PMI |
| **Inspect/metadata** | Keep lightweight (fast, avoids WASM init) | Keep lightweight |

**Build the sidecar** as a focused PMI/GD&T engine:
1. A C++ executable linked against OCCT TKXCAF + TKDESTEP (~5MB binary)
2. Accepts JSON requests: `{ "extract_pmi": { "file_path": "..." } }`
3. Returns full GD&T with shape associations (faces/edges linked to tolerances)
4. MCP server pipes requests to it transparently

This gives you the real OCCT kernel for what you need (PMI) without rewriting what already works (geometry).
