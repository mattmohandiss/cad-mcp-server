# Cadex MTK Integration Assessment Plan

## Recommendation

Integrate Cadexsoft Manufacturing Toolkit only conditionally, as an optional native or hosted deep-analysis provider. It should not replace the current portable `occt-wasm` baseline.

Cadex MTK appears useful if `cad-mcp` is moving from deterministic STEP inspection toward manufacturing intelligence: CNC feature recognition, DFM checks, semantic PMI, wall/draft analysis, sheet-metal unfolding, molding analysis, and richer assembly semantics.

## Why It Helps

- Adds manufacturing feature recognition that `occt-wasm` does not provide out of the box: holes, countersinks, pockets, milled faces, sheet-metal features, ribs, bosses.
- Adds DFM analyzers for CNC machining, sheet metal, and injection molding.
- Adds advanced measurements such as wall thickness, draft angle, and minimum distances.
- Provides semantic PMI/GD&T/annotation support that should be stronger than the current lightweight STEP text parser.
- Aligns with `docs/ROADMAP.md` Phase 5 native/deep-provider goals.

## What Not To Do

- Do not make Cadex a required dependency for `npm install`, local demos, or the current MCP server.
- Do not replace basic `inspect_step_file`, `query_step_faces`, `query_step_edges`, or `compare_step_files` behavior just to use a commercial kernel.
- Do not expose Cadex-specific classes or schemas through MCP responses.
- Do not migrate direct OCCT-backed face/edge queries during the first POC unless a specific capability requires it.

## Recommended Architecture

Keep the TypeScript MCP server as the stable protocol edge and add Cadex behind an optional provider boundary.

Preferred POC shape:

```text
MCP tools
  -> TypeScript orchestration
    -> portable providers
       -> occt-wasm
       -> lightweight STEP parser
    -> optional deep provider
       -> Cadex sidecar process or hosted worker
          -> Cadex MTK native SDK
```

The sidecar/worker should own native SDK loading, licensing, crashes, and platform-specific behavior. The MCP server should communicate through normalized JSON and continue working when Cadex is unavailable.

## POC Scope

Implement a narrow proof of concept before committing to a full integration.

Recommended POC:

- Add an optional Cadex provider/sidecar client.
- Add one experimental manufacturing feature capability, preferably CNC feature recognition.
- Optionally compare Cadex semantic PMI extraction against the existing `query_step_pmi` parser on AP242 samples.
- Keep existing tools fully functional without Cadex installed.

Candidate experimental output:

```ts
type ManufacturingFeature = {
  id: string;
  type: 'hole' | 'countersink' | 'counterbore' | 'pocket' | 'milled_face' | 'unknown';
  bodyId?: string;
  faceIds?: string[];
  bbox?: BoundingBox;
  parameters?: Record<string, number | string | boolean>;
  confidence?: number;
};
```

## Acceptance Criteria

- Baseline install, build, and tests pass without Cadex installed.
- Cadex is enabled only through explicit config or environment.
- Missing Cadex runtime returns a clear provider-unavailable result and does not crash the MCP server.
- At least one known machined STEP sample returns correct feature categories for holes plus either countersinks/counterbores or pockets.
- Feature output includes normalized IDs, type, approximate location/bbox, and useful parameters.
- If PMI is included, at least one AP242 sample demonstrates materially better semantic PMI extraction than the current text parser.
- No Cadex SDK types leak into MCP schemas.
- Runtime and memory overhead are measured on representative files.
- Licensing, redistribution, CI, and hosted-use constraints are documented before merging beyond experimental status.

## Validation Plan

- Run `just --list` first and use project recipes.
- For code changes, run `just build`, `just lint`, and `just test`.
- Add provider-unavailable tests so CI does not require Cadex.
- Add golden-output tests for any checked-in sample files that can be redistributed.
- Manually exercise the MCP tool path with Cadex disabled and, if available, enabled against the POC sample files.

## Decision Gate

Proceed beyond POC only if Cadex proves a capability that is both valuable and difficult to reproduce with `occt-wasm`: manufacturing feature recognition, DFM analysis, or semantic PMI. If the POC only improves basic B-rep inspection, do not integrate it.
