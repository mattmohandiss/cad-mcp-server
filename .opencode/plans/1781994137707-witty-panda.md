# Plan: Source cleanup before occt-wasm fork integration

## Background
Full audit of `/home/mattm/Code/cad-mcp/src/` completed (28 files). The codebase is generally clean with good architecture. Six specific issues were found — one definite bug, the rest are duplication/consistency.

## Issues by priority

### P0 — Bug (fix before fork)
**`src/providers/lightweight-step/pmi-parser.ts:72`** — `skipWhitespace()` return value ignored
- `parseParams()` calls `skipWhitespace(raw, i)` but `i` is a primitive number (passed by value, never reassigned) and the return value is discarded
- Effect: whitespace before parameter values is never skipped; fields like `'  value'` could be misparsed
- Fix: `i = skipWhitespace(raw, i);` — one line change
- Risk: near-zero; function already returns the correct value

### P1 — Unify face/edge field definitions (do before adding `axis`)
**`src/tools/step-tools.ts`** — Three overlapping face-field enums, three overlapping edge-field enums
- `FACE_GET_FIELDS`, `faceIncludeSchema`, `faceFieldsSchema` all define the same set of fields with slight name differences (`bbox_center` vs `center`)
- `EDGE_GET_FIELDS`, `edgeIncludeSchema`, `edgeFieldsSchema` — same pattern
- `adaptFaceFields` / `adaptGetFaceFields` (and edge equivalents) both do `bbox_center -> center` mapping
- Problem now: adding `axis` to face fields means editing 3 enum definitions + 2 adapter functions instead of 1
- Fix: collapse into a single source-of-truth per entity type, eliminate adapter functions

### P2 — Circular import suppression
**`src/tools/step-tools.ts:539-569`** — Three `*QuerySchema` variables declared only for `typeof` inference, suppressed with eslint-disable
- These exist because tool schemas are defined in `step-tools.ts` but consumed in `cad/query/*.ts`, creating a type-only circular import
- Already safe (type-only at runtime), but the eslint-disable is a code smell
- Fix: extract type definitions into a shared types file (or just leave it — low value)

### P3 — withErrorContext inconsistency
**`src/index.ts`** — `withErrorContext` wrapper used on 4/6 tools but not `inspect_step_file` or `compare_step_files`
- Not a bug, but inconsistent error handling behavior
- Fix: wrap the two remaining handlers for consistency (3 lines)

### P4 — Duplicate vector math
**`src/providers/occt-wasm/aag-utils.ts`** and **`src/cad/query/shared.ts`**
- Both define `dotProduct`, `magnitude`/`normalize` functions with different type signatures (Vec3 objects vs number[] arrays)
- Not blocking fork integration — both work correctly independently
- Fix: extract to `utils/vectors.ts`, import in both places

### P5 — Duplicate bbox helpers
**`src/providers/occt-wasm/query-entities.ts`** and **`src/cad/query/entities.ts`**
- `bboxToTuple` and `bboxCenter` defined twice with nearly identical logic
- Not blocking fork integration
- Fix: extract to shared location

### Won't fix (acceptable duplication)
- PMI entity type lists in `metadata.ts` vs `pmi-parser.ts` — different representations (array vs set), separate concerns
- `pmi.ts` manual grouping vs `groupEntities` in `shared.ts` — PMI entity structure differs enough that reuse would be awkward
- Test-only files are clean and well-organized
