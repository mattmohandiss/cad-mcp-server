# Docs Rewrite Plan: Portable OCCT-WASM CAD MCP

## Goal

Completely rewrite the project documentation to match the new product stance:

- Portable, local-first MCP server for read-only STEP inspection.
- TypeScript MCP server with `occt-wasm` as the default backend.
- Small, uncluttered MCP tool surface.
- AAG-style adjacency/feature analysis as an internal implementation detail, not a separate user-facing product layer.
- No OWL/SPARQL layer for now.
- Native OCCT/XDE/BRepGraph sidecar remains a future provider path, not the current product requirement.

## Key Findings

- `README.md` and `QUICK_START.md` are stale: they say AAG is unavailable, but the repo now has `OcctWasmAagProvider`.
- `Tools.md` is close in spirit but still frames the system as a broad CAD knowledge graph and points toward Analysis Situs/STP2OWL.
- The `plan/` folder describes a much broader Microsoft Copilot/SaaS/company-knowledge/digital-twin product than the current repo should represent.
- Older plans emphasize Analysis Situs and OWL/provider neutrality more than the new recommended stance.
- The current code is best documented as a portable `occt-wasm` inspection MCP with evidence-first outputs and explicit limitations.

## Recommended Documentation Structure

Keep documentation lean and accurate:

- `README.md`: product overview, current capabilities, installation, tool list, limitations, roadmap.
- `QUICK_START.md`: setup, build/test/run, sample tool calls, validation checklist.
- `Tools.md`: small public MCP surface, when to use each tool, output contracts, example prompts/tool calls.
- `docs/ARCHITECTURE.md`: portable architecture, provider boundaries, what `occt-wasm` does and does not provide, future native provider path.
- `docs/CAPABILITIES.md`: engineer-facing question coverage: reliable, candidate/heuristic, unsupported.
- `docs/ROADMAP.md`: practical phased roadmap focused on wasm-first improvements, later native provider, and distribution tiers.
- `docs/SECURITY.md`: local file handling, untrusted CAD input, MCP guardrails, future hosted/enterprise considerations.

Recommended treatment of `plan/`:

- Replace with a shorter current product roadmap, or move old content under `docs/archive/` if history should be preserved.
- Do not keep broad Copilot/company-knowledge/digital-twin docs as active guidance unless explicitly marked future/archival.

## Rewrite Content Decisions

Document these product decisions clearly:

- The MCP is read-only and does not edit CAD or generate CAM/toolpaths.
- `occt-wasm` is real upstream OCCT compiled to WebAssembly through a curated TypeScript facade.
- The MCP does not expose all upstream OCCT or BRepGraph.
- Current backend can answer geometry summary, topology, simple health, feature-candidate, report, and metric-compare questions.
- Current backend cannot promise full PMI/GD&T, stable cross-revision feature identity, native CAD feature-tree intent, robust manufacturability signoff, or full assembly semantics.
- AAG-style analysis should be described as internal face-adjacency/feature-candidate logic.
- OWL/SPARQL should be deferred until there is a concrete enterprise semantic-interoperability requirement.
- Native OCCT/XDE/BRepGraph is a future provider for production-depth analysis, not required for portable MVP.
