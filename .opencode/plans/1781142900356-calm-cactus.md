# MCP STEP Tool LLM-Friendliness And Spec Alignment Plan

## Goal

Make the CAD MCP server easier and safer for LLMs to use while keeping the server factual and data-only.

Primary outcomes:

- Keep the six-tool public surface: `inspect_step_file`, `find_step_faces`, `find_step_edges`, `get_step_entities`, `query_step_pmi`, `compare_step_files`.
- Make tool and parameter descriptions explicitly model-friendly: when to use, when not to use, and simplest examples.
- Reduce schema noise where options are not genuinely useful for common workflows.
- Make `inspect_step_file` compact and useful as the first call.
- Keep detailed groups/entities in `find_*` tools, not default inspect.
- Add MCP `outputSchema` definitions for structured results.
- Improve MCP result wrapping so successful `structuredContent` is the domain payload, and tool execution errors use `isError: true` with actionable text.
- Preserve factual-only behavior; no manufacturability judgment, feature intent inference, or issue verdicts from the server.

## Current State Summary

Relevant files:

- `src/index.ts`: registers public tools, tool descriptions, MCP `jsonToolResult`.
- `src/tools/shared.ts`: `wrapTool()` returns `{ ok: true, data }` or `{ ok: false, error }`.
- `src/tools/step-tools.ts`: public schemas, internal schemas, adapters, handlers, response interfaces.
- `src/cad/projections.ts`: inspect response projection.
- `src/cad/query/shared.ts`: shared query envelope.
- `src/cad/query/faces.ts`: face query implementation.
- `src/cad/query/edges.ts`: edge query implementation.
- `src/cad/query/pmi.ts`: PMI query implementation.
- `src/tests/*.ts`: schema, adapter, integration, PMI, MCP result tests.

Important current behavior:

- Handlers return internal `ToolResponse<T>` wrappers for direct tests.
- MCP `jsonToolResult()` currently puts the whole `{ ok, data/error }` wrapper into `structuredContent`.
- `inspect_step_file` currently returns `filePath`, `facts`, and full face/edge grouped summaries with sample IDs by default.
- Query responses currently include an internal-ish `query` echo, not a clean public `requested`/`applied` view.
- Public schemas are flat and reject old nested keys. Preserve this.

## Recommended Public Tool Surface

Keep exactly these six public tools:

1. `inspect_step_file`
2. `find_step_faces`
3. `find_step_edges`
4. `get_step_entities`
5. `query_step_pmi`
6. `compare_step_files`

Do not add a collection/review tool. The LLM should use `inspect_step_file` first, then drill down with find/get tools.

## MCP Result Handling

### Internal Handler Contract

Keep direct handler return values as:

```ts
type ToolResponse<T> =
  | { ok: true; data: T }
  | { ok: false; error: { type: string; message: string } };
```

Reason: existing tests and internal error normalization already depend on this. It is useful below the MCP boundary.

### MCP Boundary Contract

Change `jsonToolResult()` in `src/index.ts`:

- If input is `{ ok: true, data }`:
  - `structuredContent = data`
  - text content is `JSON.stringify(data, null, 2)`
  - `isError = false`
- If input is `{ ok: false, error }`:
  - no successful `structuredContent`
  - text content is an actionable error string including type and message
  - `isError = true`
- If input is raw payload, preserve current fallback behavior for tests/utilities if needed.

This aligns better with MCP: the model sees the domain payload directly on success and an actionable tool execution error on failure.

Update `src/tests/mcp-result.test.ts` accordingly.

## Output Schemas

Add `outputSchema` to every `registerTool()` config in `src/index.ts`.

Implementation approach:

- Define reusable output schema shapes in `src/tools/step-tools.ts`, exported as `stepToolOutputSchemas`.
- Keep output schemas practical and top-level, not exhaustive CAD entity schemas. The goal is client/LLM guidance and validation of major response structure, not every nested numeric property.
- Register each tool with matching `outputSchema`.

Recommended output schema exports:

- `inspectStepFile`: compact inspect response schema.
- `findStepFaces`: shared query response schema.
- `findStepEdges`: shared query response schema.
- `getStepEntities`: shared query response schema.
- `queryStepPmi`: shared query response schema.
- `compareStepFiles`: compare response schema.

Register type in `src/index.ts` should accept:

```ts
outputSchema?: Record<string, z.ZodType>;
```

### 2. `find_step_faces`

Tool description:

```text
Search faces by surface type, area, spatial region, or proximity. Use when you need specific faces such as large faces, cylindrical faces, faces near a point, or grouped face statistics. Do NOT use for known face IDs; use get_step_entities. Use only the filters needed for the question; adding every optional filter can over-constrain results. Example small faces: {file_path:"model.step",area_max:1,sort_by:"area"}. Example cylinders by radius: {file_path:"model.step",surface_types:["cylinder"],return_type:"groups",group_by:["radius"]}
```

Input schema parameters:

```text
file_path: string required
surface_types?: enum[] plane|cylinder|cone|sphere|torus|bspline|other
area_min?: number >= 0
area_max?: number >= 0
normal_parallel_to?: [number, number, number]
bbox_min?: [number, number, number]
bbox_max?: [number, number, number]
center_near_point?: [number, number, number]
center_near_distance?: number >= 0
body_ids?: body:N[]
fields?: enum[] id|surface_type|area|bbox|bbox_center|normal|surface_parameters|adjacent_faces|closest_face_distance|has_inner_wires|body_id
group_by?: enum[] surface_type|area_range|radius
sort_by?: enum area
sort_direction?: asc|desc
return_type?: summary|entities|groups
limit?: integer 1..1000
offset?: integer >= 0
sample_entity_limit?: integer 0..50
```

Schema reductions from current:

- Remove public `normal_tolerance_degrees`; use internal default tolerance of 15 degrees when `normal_parallel_to` is supplied.
- Remove public `bbox_match`; use internal `intersects` mode for `bbox_min`/`bbox_max`.
- Remove `normal_direction` from public `group_by`.
- Restrict public `sort_by` to `area`.

Parameter description guidance:

- `surface_types`: “Omit to include all surface types.”
- `normal_parallel_to`: “Omit unless you need directional filtering; matches within 15 degrees.”
- `bbox_min` / `bbox_max`: “Must provide both or neither; uses intersecting bbox match.”
- `center_near_point` / `center_near_distance`: “Must provide both or neither.”
- `body_ids`: “Only needed for multi-body files; omit for all bodies.”
- `group_by`: “Requires `return_type: "groups"`; ignored/noticed otherwise.”

Validation:

- Reject `face_ids` as unknown.
- `area_min <= area_max` when both are present.
- bbox and center-near pairs must be complete.
- non-zero normal vector.

### 3. `find_step_edges`

Tool description:

```text
Search edges by curve type, length, circular radius, spatial region, or proximity. Use for tiny-edge investigation, long edges, circular edges, or grouped edge statistics. For tiny edges, use length_max and sort_by:"length"; omit radius filters unless specifically querying circular edges by radius. radius_min/radius_max only affect circle edges and do not exclude line or bspline edges. Do NOT use for known edge IDs; use get_step_entities. Example tiny edges: {file_path:"model.step",length_max:0.5,sort_by:"length",fields:["id","length","curve_type"]}
```

Input schema parameters:

```text
file_path: string required
curve_types?: enum[] line|circle|ellipse|bspline|other
length_min?: number >= 0
length_max?: number >= 0
radius_min?: number >= 0
radius_max?: number >= 0
bbox_min?: [number, number, number]
bbox_max?: [number, number, number]
center_near_point?: [number, number, number]
center_near_distance?: number >= 0
body_ids?: body:N[]
fields?: enum[] id|curve_type|length|bbox|bbox_center|radius|start_point|end_point|adjacent_faces|body_id
group_by?: enum[] curve_type|length_range
sort_by?: enum length|radius
sort_direction?: asc|desc
return_type?: summary|entities|groups
limit?: integer 1..1000
offset?: integer >= 0
sample_entity_limit?: integer 0..50
```

Schema reductions from current:

- Remove public `bbox_match`; use internal `intersects` mode.
- Restrict public `sort_by` to `length | radius`.

Behavior fix:

- `radius_min` / `radius_max` must only apply to circular edges.
- Non-circular edges should not be excluded solely because a radius filter is present.
- Add tests for `length_max` plus broad radius filters returning short line/bspline edges when applicable.

Validation:

- Reject `edge_ids` as unknown.
- `length_min <= length_max` when both are present.
- `radius_min <= radius_max` when both are present.
- bbox and center-near pairs must be complete.

### 4. `get_step_entities`

Tool description:

```text
Retrieve one or more known faces or edges by exact entity ID. IDs come from inspect_step_file, find_step_faces, or find_step_edges. Use ONLY when you already have specific IDs. Do NOT use for searching or filtering; use find_step_faces or find_step_edges. Example: {file_path:"model.step",entity_type:"face",entity_ids:["face:0"],fields:["id","area","normal"]}
```

Input schema parameters:

```text
file_path: string required
entity_type: face|edge required
entity_ids: string[] required, 1..200, unique, prefix must match entity_type
fields?: enum[] id|surface_type|area|curve_type|length|bbox|bbox_center|normal|surface_parameters|radius|start_point|end_point|adjacent_faces|closest_face_distance|has_inner_wires|body_id
```

Keep current behavior:

- Validate `face:N` IDs for faces and `edge:N` IDs for edges.
- Validate requested fields against entity type.

### 5. `query_step_pmi`

Tool description:

```text
Query Product Manufacturing Information (PMI): geometric tolerances, dimensions, datums, and annotations. Use return_type:"summary" first to check whether PMI exists. Not all STEP files contain PMI; AP203 files commonly do not. Example quick check: {file_path:"model.step",return_type:"summary"}
```

Input schema parameters:

```text
file_path: string required
pmi_types?: enum[] geometric_tolerance|dimension|datum|annotation
tolerance_subtypes?: enum[] position|flatness|straightness|circularity|cylindricity|profile|parallelism|perpendicularity|angularity|concentricity|runout|symmetry|coaxiality|circular_runout|total_runout|surface_profile|line_profile
value_min?: number >= 0
value_max?: number >= 0
group_by?: enum[] type|tolerance_type|dimension_type|material_condition
sort_by?: enum type|value|tolerance_type
sort_direction?: asc|desc
return_type?: summary|entities|groups
limit?: integer 1..1000
offset?: integer >= 0
sample_entity_limit?: integer 0..50
```

Keep schema mostly unchanged; improve descriptions only.

### 6. `compare_step_files`

Tool description:

```text
Compare two STEP files and return whole-model metric deltas and metadata changes. Use for two revisions of a part when you need factual differences in dimensions, volume, area, topology counts, or exchange metadata. Does NOT track feature identity across revisions. Deltas are comparison_file_path minus baseline_file_path. Example: {baseline_file_path:"model_v1.step",comparison_file_path:"model_v2.step"}
```

Input schema parameters:

```text
baseline_file_path: string required
comparison_file_path: string required
```

Keep current behavior. Optional future improvement: add surface type and curve type count deltas, but do not include in this change unless cheap and testable.

## Shared Query Response Envelope

Modify shared query response from current `query` echo to this shape:

```json
{
  "schema_version": "0.4",
  "file_path": "model.step",
  "units": { "length": "mm", "area": "mm^2", "volume": "mm^3", "angle": "deg" },
  "coordinate_system": { "origin": "STEP model origin", "axes": "model coordinates", "handedness": "right" },
  "requested": { "length_max": 1, "sort_by": "length" },
  "applied": {
    "filters": { "length_max": 1 },
    "fields": ["id", "curve_type", "length", "bbox_center"],
    "return_type": "entities",
    "sort": { "by": "length", "direction": "asc" },
    "pagination": { "limit": 100, "offset": 0 }
  },
  "notices": [],
  "diagnostics": [],
  "statistics": {},
  "pagination": {},
  "entities": [],
  "groups": [],
  "warnings": [],
  "limitations": []
}
```

Rules:

- `requested` should echo only user-supplied public parameters, excluding `file_path`.
- `applied` should use public names, not internal names. For example, use `bbox_center`, not `center`.
- If `group_by` is supplied with `return_type: "entities"`, do not fail; add notice that `group_by` is ignored unless `return_type` is `groups`.
- Add `diagnostics` when zero matches occur and a likely over-constraint exists. Example: if edge query has `matched_edges: 0` and radius filters were supplied, include “No edges matched; remove radius filters unless searching circular edges by radius.”
- Preserve `statistics`, `pagination`, `entities`, and `groups` fields for stable client use.

Implementation note:

- Internal query services can still use nested `filter`, `region`, `near`, `include`, and `result_mode`.
- Public handlers/adapters should pass enough metadata to response assembly to populate `requested` and `applied`.
- If this is too invasive in one pass, preserve existing `query` internally but transform it to public `requested`/`applied` at response boundary.

## Description Quality Rules

Apply these consistently in `src/index.ts` and `src/tools/step-tools.ts`:

- Tool descriptions: 3-5 concise sentences with purpose, when not to use, and one example.
- Parameter descriptions: concise and action-oriented.
- Mention when to omit a parameter.
- Avoid internal words in public descriptions: no `include`, `result_mode`, `center` where public term is `fields`, `return_type`, `bbox_center`.
- Fix inaccurate wording: arrays of enum filters use OR logic, not AND.

## Non-Goals

- No manufacturability verdicts.
- No feature recognition or design-intent inference.
- No new public tools.
- No backward compatibility with old public schemas unless tests or user requirements demand it.
- No committing changes unless explicitly requested.
