# Architecture Plan: Provider-Neutral CAD Knowledge Graph

## Goal
Refactor `cad-mcp` around a provider-neutral CAD knowledge architecture. The MCP should expose five AI-facing STEP tools, while the implementation remains free to swap B-rep, AAG, and semantic/OWL providers over time.

The core idea:

```text
BRepProvider -> BRepModel
AagProvider -> AagModel
SemanticProvider -> SemanticModel
  -> GraphBuilder
  -> CadKnowledgeGraph
  -> MCP tool projections
```

## Architectural Decisions
- Use three separate layer models: `BRepModel`, `AagModel`, and `SemanticModel`.
- Merge those layer models into one canonical `CadKnowledgeGraph` in the application layer.
- Design provider interfaces before committing to concrete dependencies.
- Keep public MCP tools thin and independent of provider implementation details.
- Replace old public tools (`analyze_step_file`, `list_bodies`, `extract_edges`) with the five-tool surface from `Tools.md`.
- Treat B-rep, AAG, and semantic/OWL as stable internal concepts, not vendor lock-in.
- Return `generate_step_report` as structured JSON plus Markdown.
- Keep outputs evidence-first: facts, inferences, warnings, limitations, and provenance.

## Non-Goals
- Do not implement direct geometry editing.
- Do not generate CAM/toolpaths.
- Do not claim authoritative tolerance/inspection facts unless explicit PMI supports them.
- Do not make off-the-shelf AI infer features from raw STEP or raw B-rep dumps.
- Do not lock the architecture to `occt-wasm`, Analysis Situs, AAGNet, STP2OWL, Neo4j, RDF, or any one graph library.

## Dependency Direction

```text

```

Avoid:
- `tools` importing `occt-wasm`, Analysis Situs, AAGNet, or STP2OWL directly.
- `cad` importing MCP SDK types.
- provider implementations importing tool handlers.
- `utils` importing CAD domain modules.

## Provider Interfaces

### `BRepProvider`
Purpose: exact geometry and topology facts from STEP or another CAD source.

Responsibilities:
- load/import file
- expose bodies, shells, faces, loops, edges, vertices where available
- compute measurements
- report topology/import health
- emit provider-neutral IDs and provenance

Shape:

```ts
interface BRepProvider {
  name: string;
  capabilities: BRepCapability[];
  load(filePath: string, options?: BRepLoadOptions): Promise<BRepModel>;
}
```

Initial implementation:
- `OcctWasmBRepProvider` using existing `occt-wasm` functionality.

Future implementations:
- Analysis Situs/OpenCascade sidecar
- CAD Exchanger or other CAD kernel
- native OpenCascade service

### `AagProvider`
Purpose: face-adjacency and feature-recognition graph derived from B-rep data.

Responsibilities:
- build AAG nodes and arcs
- attach face/arc attributes such as surface type, area, shared edges, vexity, dihedral angle
- optionally recognize engineering/manufacturing feature candidates
- emit evidence and source IDs for each feature candidate

Shape:

```ts
interface AagProvider {
  name: string;
  capabilities: AagCapability[];
  build(input: AagInput, options?: AagOptions): Promise<AagModel>;
  recognizeFeatures?(input: AagModel, options?: FeatureOptions): Promise<FeatureCandidate[]>;
}
```

Initial implementation:
- Minimal fallback AAG from available `occt-wasm` topology if practical.
- If true face adjacency is not available from `occt-wasm`, return a limited AAG section with explicit limitations.

Preferred advanced implementation:
- Analysis Situs sidecar for AAG JSON and deterministic feature recognizers.

Optional later implementation:
- AAGNet sidecar for ML-backed machining feature recognition.

### `SemanticProvider`
Purpose: exchange metadata and semantic/OWL-like facts.

Responsibilities:
- parse STEP header/schema/protocol
- detect product names and authoring hints where possible
- detect PMI/entity keyword presence with limitations
- optionally export/import OWL/RDF-compatible facts later

Shape:

```ts
interface SemanticProvider {
  name: string;
  capabilities: SemanticCapability[];
  extract(filePath: string, options?: SemanticOptions): Promise<SemanticModel>;
  export?(graph: CadKnowledgeGraph, options?: SemanticExportOptions): Promise<SemanticExport>;
}
```

Initial implementation:
- `LightweightStepSemanticProvider` using lightweight STEP text/header parsing.

Future implementation:
- STP2OWL sidecar/export path for formal OWL representation.

## Canonical Layer Models

### `BRepModel`
Represents measured and topological source facts.

Core entities:
- `BRepBody`
- `BRepShell`
- `BRepFace`
- `BRepLoop`
- `BRepEdge`
- `BRepVertex`
- `Measurement`
- `BRepHealth`

Minimum first-pass facts:
- file path
- units
- body count
- bounding box
- dimensions
- volume
- surface area
- face count if available
- edge count if available
- edge statistics if available
- body metrics
- import/topology health warnings

### `AagModel`
Represents the derived face-adjacency layer.

Core entities:
- `AagNode`: face-backed graph node
- `AagEdge`: face-adjacency arc
- `FeatureCandidate`: inferred engineering/manufacturing feature

Minimum first-pass facts:
- face nodes where available
- adjacency arcs where available
- surface/curve attributes where available
- feature candidates for holes/fillets from current heuristics or explicit limited results
- limitations if provider cannot compute true AAG

### `SemanticModel`
Represents exchange/standards/OWL-like facts.

Core entities:
- schema/protocol metadata
- units metadata
- product names
- authoring system hints
- PMI summary
- validation-property hints
- semantic facts
- limitations

Minimum first-pass facts:
- STEP schema/header detection
- AP/schema string if present
- product/name strings if lightweight parsing can find them
- PMI keyword/entity presence, clearly marked as heuristic
- limitations explaining that semantic PMI/GD&T is not fully parsed

## Canonical `CadKnowledgeGraph`
The graph builder merges provider outputs into one AI-facing domain graph.

Core graph entities:
- `CadNode`
- `CadEdge`
- `Fact`
- `Inference`
- `Warning`
- `Limitation`
- `Evidence`
- `Provenance`

Node categories:
- `file`
- `body`
- `shell`
- `face`
- `loop`
- `edge`
- `vertex`
- `measurement`
- `aag_face`
- `feature_candidate`
- `spatial_relationship`
- `exchange_metadata`
- `pmi_item`
- `warning`
- `limitation`

Edge categories:
- `contains`
- `bounded_by`
- `adjacent_to`
- `measured_by`
- `evidence_for`
- `warns_about`
- `derived_from`
- `annotates`
- `related_to`

Evidence shape:

```ts
interface Evidence {
  confidence: number;
  sourceIds: string[];
  provider: string;
  method: 'measured' | 'derived' | 'heuristic' | 'ml' | 'semantic';
  explanation: string[];
  limitations: string[];
}
```

## Public MCP Tools

### `inspect_step_file`
Fast first-pass overview.

Uses:
- B-rep summary
- semantic metadata summary
- health/risk summary
- high-level graph facts

Returns:
- geometry summary
- structure summary
- exchange summary
- health warnings
- complexity headline
- limitations

### `analyze_step_detail`
Deep category-selected analysis.

Input categories:
- `geometry`
- `topology`
- `structure`
- `features`
- `spatial`
- `exchange`
- `health`

Detail levels:
- `summary`
- `standard`
- `full`

Returns:
- graph projections for requested categories
- evidence-rich feature candidates
- provider limitations

### `query_step_graph`
Deterministic graph query interface.

First-pass query support:
- find nodes by type/category
- filter feature candidates by type/confidence/dimension where available
- retrieve relationships between node IDs
- retrieve evidence for a node/inference
- retrieve warnings/limitations by severity/category

Do not implement LLM-generated SPARQL or arbitrary code execution.

### `compare_step_files`
Graph/snapshot comparison between two STEP files.

First-pass comparison:
- dimensions/bounding-box deltas
- volume/surface-area deltas
- body/face/edge count deltas
- feature candidate count deltas
- exchange metadata deltas
- health/risk deltas

Avoid claiming stable feature identity across revisions until robust matching exists.

### `generate_step_report`
Human-readable report from graph facts.

Returns:
- structured JSON sections
- Markdown report

Report types:
- `engineering_review`
- `supplier_review`
- `import_risk`
- `space_claim`
- `manufacturing_handoff`
- `pmi_audit`

## AI-Facing Output Rules
Every public tool should separate:
- `facts`: directly measured or parsed values
- `inferences`: feature/spatial/health candidates with evidence
- `warnings`: risks and suspicious conditions
- `limitations`: what cannot be proven from current providers/STEP data
- `provenance`: provider and source ID information

The MCP should give off-the-shelf AI models compact engineering facts, not raw STEP or huge B-rep dumps by default.

## Dependency Strategy

### First Implementation Dependencies
Use existing dependencies only:
- `@modelcontextprotocol/sdk`
- `zod`
- `occt-wasm`

No new runtime dependency is required for the initial provider-interface/type-system refactor.

### Planned Optional Providers
- Analysis Situs sidecar: preferred advanced AAG and deterministic feature-recognition backend.
- AAGNet sidecar: optional ML machining-feature recognizer.
- STP2OWL sidecar/exporter: optional formal OWL/semantic STEP representation.

### Dependency Rule
Add a runtime dependency only when it provides one of:
- CAD kernel capability we cannot reasonably implement.
- Correct standard parsing we would otherwise do poorly.
- Measured performance improvement.
- Major reduction in risky feature-recognition code.

## Implementation Stages

### Stage 1: Type System And Boundaries
- Create provider interfaces and canonical schemas.
- Define `BRepModel`, `AagModel`, `SemanticModel`, and `CadKnowledgeGraph`.
- Define evidence/provenance/warning/limitation types.
- Add graph utility functions for typed node/edge lookup.

### Stage 2: Current Functionality As Providers
- Move current `occt-wasm` logic behind `OcctWasmBRepProvider`.
- Add safe STEP import lifecycle with cleanup in `finally`.
- Implement lightweight semantic provider from STEP text/header parsing.
- Implement limited AAG provider or explicit limited AAG result from current capabilities.

### Stage 3: Graph Builder
- Merge B-rep, AAG, and semantic models into `CadKnowledgeGraph`.
- Generate stable typed IDs.
- Preserve provider provenance.
- Promote measured values to facts.
- Promote heuristics to inferences with confidence and limitations.

### Stage 4: Five Tool Surface
- Replace tool registration with the five tools.
- Add `src/tools/shared.ts` response helpers.
- Implement each tool as a projection/query/report over `CadKnowledgeGraph`.
- Remove old tool handlers or stop registering them.

### Stage 5: Tests
- Move fixture helpers to `src/tests/fixtures.ts`.
- Update integration tests to cover the five tools.
- Add tests for provider outputs, graph build shape, graph query behavior, comparison deltas, and JSON-plus-Markdown report output.

### Stage 6: Optional Advanced Provider Planning
- Specify Analysis Situs sidecar JSON contract.
- Specify AAGNet sidecar contract if ML feature recognition becomes a priority.
- Specify STP2OWL import/export contract if formal OWL becomes a priority.

## Final Architecture Clarifications
- If current `occt-wasm` cannot produce true face adjacency/vexity, mark AAG as unavailable with explicit limitations until Analysis Situs or another AAG provider is integrated. Do not fake a full AAG.
- Graph IDs only need to be stable within a single analysis run in the first implementation. Revision-stable IDs can be added later when robust matching exists.
- Every public tool response should include a concise provider/capability/limitation summary. Full detail responses can include expanded capability diagnostics.
