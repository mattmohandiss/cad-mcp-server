# Refactor Plan: Remove Redundancy And Reduce CAD Overhead

## Context
- Source surface is small: `src/index.ts`, three tool handlers, `src/utils/occt-cad.ts`, `src/utils/schema.ts`, one re-export file, and integration tests.
- The main runtime cost is in `src/utils/occt-cad.ts`: every tool call reads and imports the STEP file independently, imported OCCT shapes are not released, feature detection repeats topology traversal, and edge extraction materializes data the public handler mostly discards.
- The main code redundancy is repeated MCP registration/result handling, repeated success/error envelopes, duplicated schema structures, scattered numeric formatting, and duplicate shape metric logic.
- User confirmed public output changes are allowed and performance/simplification should be balanced equally.

## Recommended Approach

### 1. Normalize Tool Results And Registration
Modify:
- `src/index.ts`
- `src/tools/analyze.ts`
- `src/tools/bodies.ts`
- `src/tools/edges.ts`
- add `src/tools/shared.ts` if it keeps the handlers smaller

Plan:
- Define a single tool response shape: `{ ok: true, data }` and `{ ok: false, error: { type, message } }`.
- Centralize `AnalysisError` conversion in one helper instead of repeating `success: false` blocks in all handlers.
- Register MCP tools from a small definition array in `src/index.ts` to remove repeated `registerTool` boilerplate.
- Keep summaries only if useful, but make structured numeric fields the canonical output.

### 2. Consolidate STEP Import Lifecycle
Modify:
- `src/utils/occt-cad.ts`

Plan:
- Replace `access(filePath) + readFile(filePath)` with `readFile(filePath)` and existing user-facing error mapping.
- Add a `withImportedStep(filePath, action, fn)` helper that owns: read file, initialize singleton kernel, import STEP, run analysis, map OCCT errors, and cleanup in `finally`.
- Keep the singleton `OcctKernel`; kernel startup is expensive and not the main problem.
- Release imported shapes after each request using the package-supported cleanup API. The installed docs state shapes are freed when the kernel is disposed or `release()` is called, but implementation should confirm the exact callable signature before editing.
- Do not implement raw shape caching in the first pass. It would retain native/WASM handles and make release ownership harder.

### 3. Extract Shared Geometry Types And Helpers
Modify:
- `src/utils/schema.ts`
- `src/utils/occt-cad.ts`

Plan:
- Extract shared types such as `Point3D`, `BoundingBox`, `Dimensions`, `DetectedFeatures`, `ShapeMetrics`, and `AnalysisResult<T>`.
- Add helpers for `toBoundingBox`, `getDimensions`, `getShapeMetrics`, `getBodies`, and `guessShapeClass`.
- Use those helpers from both file-level and body-level analysis.
- Return numbers for volumes, surface areas, and lengths everywhere. Remove handler-level `.toFixed()` fields from the JSON payload.
- Add a `units` object where useful, for example `{ length: 'mm', area: 'mm^2', volume: 'mm^3' }`.

### 4. Refactor Analyze And Body Paths Around Shared Metrics
Modify:
- `src/utils/occt-cad.ts`
- `src/tools/analyze.ts`
- `src/tools/bodies.ts`

Plan:
- Make `analyzeStepFile(filePath)` return one structured file-level object with `filePath`, `bodyCount`, `shapeType`, `metrics`, `features`, and `units`.
- Make `listBodies(filePath)` return `{ filePath, bodyCount, bodies, units }`, where each body has `index`, `metrics`, and `features`.
- Use the same metric helper for root shapes and body shapes.
- Keep feature detection, but make it short-circuit once it has enough evidence for the booleans.

### 5. Optimize Edge Extraction Into One Pass
Modify:
- `src/utils/occt-cad.ts`
- `src/tools/edges.ts`

Plan:
- Stop returning the full `edges` array by default because the public handler only needs aggregate stats and ranges.
- Compute edge count, total length, average/min/max length, curve type counts, and length buckets in a single pass over `kernel.getSubShapes(shape, 'edge')`.
- Replace `Math.min(...lengths)` and `Math.max(...lengths)` with incremental min/max to avoid large argument spreads.
- Reuse edge traversal data for edge-derived features so `detectFeatures` does not call `getSubShapes(shape, 'edge')` again.
- Return a compact structured result, for example `edges: { count, totalLength, averageLength, minLength, maxLength, byCurveType, byLengthRange }`.

### 6. Remove Dead Indirection And Test Redundancy
Modify:
- `src/utils/cad-analyzer.ts`
- imports in `src/tools/*.ts`
- `src/tests/integration.test.ts`

Plan:
- Remove `src/utils/cad-analyzer.ts` if no package/API contract depends on it; import directly from `src/utils/occt-cad.ts`.
- Update integration tests for the new `{ ok, data }` result shape and numeric payloads.
- Simplify test helpers to assert success/error once.
- Optionally initialize one fixture-generation kernel in `beforeAll` and dispose it once, instead of starting an OCCT kernel for every generated fixture.

### 7. Defer Caching Until After Baseline Cleanup
Plan:
- Do not add raw OCCT shape caching in the initial refactor.
- If repeated same-file calls are still too slow after cleanup, add a bounded result cache keyed by resolved file path, size, mtime, and tool/result type.
- Prefer caching plain JS results over shape handles to avoid WASM arena ownership issues.

## Product Research Addendum: Engineer-Facing STEP Reasoning Tools

### Practitioner Themes
- Mechanical engineers often receive large vendor/OEM STEP assemblies that are too detailed for their immediate task. Forum anecdotes repeatedly mention hundreds or thousands of bodies, excessive faces, screws/bolts/washers, cosmetic threads, tiny fillets, labels, internal cavities, and face-level appearances causing CAD slowdowns.
- Engineers commonly need only a subset of a vendor model: envelope/space claim, mounting holes, inlet/outlet ports, mass properties, recognizable external shape, and enough geometry for interference checks.
- STEP import is often a triage problem before it is a modeling problem: users want to know whether a file is solid vs surface-heavy, assembly vs multibody, missing/broken geometry, huge due to small details, or likely to crash/freeze CAD.
- Cross-functional exchange still depends heavily on STEP plus PDF drawings and sometimes Excel BOMs. STEP alone usually does not carry the full design intent, tolerances, material, or supplier context that downstream users need.
- Engineers are wary of black-box automation for inspection/manufacturing. Read-only tools should explain what they measured, what assumptions they made, and what data is absent from the STEP.
- For assemblies, users care about hierarchy, duplicate/repeated components, part counts, body counts, file size drivers, and whether components are internal/small enough to suppress or ignore for layout/review.

### Useful Read-Only MCP Tool Ideas
- `inspect_step_structure`: report STEP protocol/header info, product names, assembly depth, instance count, body count, solid/surface counts, face/edge counts, colors/metadata/PMI presence where detectable.
- `summarize_geometry_complexity`: produce a complexity score with drivers such as body count, face count, edge count, curved face count, tiny edge count, surface-body ratio, and bounding-box/file-size ratios.
- `find_simplification_candidates`: identify likely removable details without editing the model: tiny bodies, fastener-like cylinders, cosmetic text/markings, tiny fillets/chamfers, internal bodies, internal cavities, and repeated small components.
- `analyze_space_claim`: return bounding box, oriented or axis-aligned envelope if feasible, volume, center of mass, mounting-hole candidates, port/cylinder candidates, and external-size summary for layout/interference reasoning.
- `list_assembly_inventory`: return a structured component/body inventory with names if present, approximate dimensions, volume, surface area, location/bounding box, repeated geometry hints, and likely role tags such as fastener, plate, tube, bracket, housing, or unknown.
- `check_step_health`: flag import and topology risks such as open shells, non-solid/surface-only content, unusually short edges, zero/near-zero area faces, high face counts, many disconnected bodies, and missing/duplicate names.
- `compare_step_files`: compare two STEP revisions by bounding boxes, body/component counts, volume, surface area, centroid, and major changed regions. This helps version/release review without opening full CAD.
- `generate_supplier_review_report`: create a concise read-only report for quoting/review: dimensions, mass-property availability, component counts, likely missing drawing/PMI data, complexity risks, and suggested questions for the supplier.
- `extract_manufacturing_features`: detect candidate holes, cylinders, planar faces, thickness-like dimensions, bends/sheet-metal hints, threads/cosmetic-thread hints, and feature counts. This should be advisory, not authoritative.
- `estimate_import_risk`: predict whether a CAD import will be heavy based on file size, topology counts, surface-body count, face-level metadata, and assembly breadth; include practical recommendations such as request Parasolid/native/JT, ask supplier for simplified/envelope STEP, or import selectively.

### Product Direction Implications
- Keep the MCP read-only and explanatory. Engineers may trust tools that inspect and advise more readily than tools that mutate geometry.
- Optimize for quick triage of large STEP files before CAD import: “What is in this?”, “Why is it heavy?”, “What can I ignore?”, “What should I ask the supplier for?”, and “Is this safe for layout/quote/inspection use?”
- Prefer structured numeric data plus human-readable summaries. Engineers and agents need both machine-readable fields and plain-language risk explanations.
- Include confidence/limitations in outputs, especially for inferred features such as holes, fasteners, internal bodies, and manufacturability hints.
- After the refactor, add tools incrementally starting with complexity/health/inventory because they reuse the same topology traversal improvements and provide immediate value without editing CAD geometry.

## Revised `Tools.md` Direction

### Purpose
`Tools.md` should be the product and architecture guide for the STEP MCP tool surface. It should explain that this MCP is a read-only STEP knowledge base for mechanical-design reasoning, not a pile of disconnected CAD utilities, CAD editor, CAM generator, or replacement for drawings/native CAD.

### Best Mental Model
The MCP should expose a few access patterns over one canonical CAD knowledge graph:
- inspect
- analyze
- query
- compare
- report

The seven master categories are the actual knowledge domains.

### Master Categories

#### 1. Geometry
Measured physical properties.

Answers:
- What size is it?
- What is its envelope?
- What are its mass-like properties?
- What are the main bodies?

Includes:
- bounding box
- dimensions
- volume
- surface area
- centroid / center of mass
- body count
- face/edge counts
- units
- largest/smallest bodies
- envelope / space claim

#### 2. Topology
How the B-rep is connected.

Answers:
- Which faces touch?
- Which edges bound which faces?
- Is this a closed solid?
- Where are concave/convex relationships?

Includes:
- bodies, shells, faces, loops, edges, vertices
- face adjacency graph
- edge-face relationships
- concave/convex/smooth adjacency
- inner loops
- open/naked edges
- non-manifold edges
- topology graph / AAG

#### 3. Structure
How the STEP file is organized as a part or assembly.

Answers:
- Is this a part, multibody part, or assembly?
- What components exist?
- Are there repeated instances?
- What names/products did the supplier include?

Includes:
- assembly hierarchy
- component prototypes
- component instances
- transforms/locations
- repeated parts
- body-to-component mapping
- product names
- subassembly depth
- instance counts

#### 4. Features
Engineering-relevant shape candidates inferred from geometry/topology.

Answers:
- Are there hole-like features?
- Are there fillets/chamfers?
- Are there pockets, slots, ribs, bosses, shafts?
- What features might matter for machining or mounting?

Includes:
- hole candidates
- mounting-hole candidates
- pocket candidates
- slot candidates
- cavity candidates
- fillet candidates
- chamfer candidates
- boss/shaft candidates
- rib candidates
- port candidates
- fastener-like bodies
- feature dimensions
- confidence/evidence/limitations

#### 5. Spatial Relationships
Relationships between bodies, faces, components, and features.

Answers:
- Are these holes aligned?
- Is this face parallel to that one?
- Which bodies are internal?
- What defines the outer envelope?
- Are features patterned or symmetric?

Includes:
- coaxial
- concentric
- coplanar
- parallel
- perpendicular
- near/far
- inside/outside
- intersects bounding box
- aligned with
- repeated pattern
- symmetry candidates
- envelope-defining components

#### 6. Exchange And Standards
What the STEP file carries as exchange data.

Answers:
- What STEP protocol is this?
- Does it include PMI?
- Are tolerances/dimensions/datum references present?
- Is PMI semantic or only graphical?
- Are validation properties present?

Includes:
- AP203/AP214/AP242/etc.
- schema/header metadata
- authoring system if present
- units
- product names
- semantic PMI
- graphical PMI
- GD&T
- dimensions
- datums
- saved views
- validation properties
- precise B-rep vs tessellated geometry
- AP242/MBD readiness

#### 7. Health, Complexity, And Risk
How usable, heavy, or suspicious the file is.

Answers:
- Will this be painful to import?
- Why is the file heavy?
- Is it surface-heavy or solid?
- What should I ask the supplier to simplify?
- What should I verify before manufacturing?

Includes:
- import success/failure
- invalid geometry
- surface-only content
- open shells
- tiny edges/faces
- high face count
- high body count
- excessive small components
- likely cosmetic detail
- missing names
- duplicate names
- complexity score
- import risk
- simplification candidates
- missing PMI/drawing warnings

### Public Tool Surface
Keep the actual MCP tools small.

#### 1. `inspect_step_file`
Fast first-pass overview.

Covers:
- geometry summary
- structure summary
- exchange metadata
- basic health warnings
- complexity headline

Use when:
- user first asks about a file
- AI needs initial context

#### 2. `analyze_step_detail`
Deep analysis with selectable categories.

Example input:
```json
{
  "file_path": "model.step",
  "categories": [
    "geometry",
    "topology",
    "structure",
    "features",
    "spatial",
    "exchange",
    "health"
  ],
  "detail_level": "summary"
}
```

Use when:
- user asks deeper questions
- AI needs features, graph, topology, or health details

#### 3. `query_step_graph`
Targeted graph queries across turns.

Example inputs:
```json
{
  "file_path": "model.step",
  "query": {
    "find": "features",
    "where": {
      "type": "hole_candidate",
      "diameter_gte": 5
    }
  }
}
```

```json
{
  "file_path": "model.step",
  "query": {
    "find": "relationships",
    "between": ["feature:hole_candidate:1", "feature:hole_candidate:4"]
  }
}
```

Use when:
- user asks specific follow-up questions
- AI needs precise subgraph facts

#### 4. `compare_step_files`
Revision or supplier-file comparison.

Covers:
- geometry deltas
- structure deltas
- feature deltas
- metadata/PMI deltas
- health/risk deltas

Use when:
- user asks “what changed?”
- supplier sends a new version

#### 5. `generate_step_report`
Human-readable synthesis from graph facts.

Report types:
- `engineering_review`
- `supplier_review`
- `import_risk`
- `space_claim`
- `manufacturing_handoff`
- `pmi_audit`

Use when:
- user wants a concise report
- AI needs to communicate findings to another engineer/supplier

### Internal Knowledge Graph
All tools should draw from one canonical graph:

```text
CAD Knowledge Graph
  geometry
  topology
  structure
  features
  spatial relationships
  exchange/PMI
  health/risk
  evidence/provenance
```

Use a heterogeneous attributed property graph with stable IDs and typed nodes/edges.

Core node types:
- `assembly`, `component_prototype`, `component_instance`, `body`, `shell`, `face`, `loop`, `edge`, `vertex`, `feature_candidate`, `warning`, `measurement`, `pmi_item`.

Core edge types:
- `contains`, `instantiates`, `transformed_by`, `bounded_by`, `adjacent_to`, `consists_of`, `evidence_for`, `coaxial_with`, `coplanar_with`, `parallel_to`, `perpendicular_to`, `repeated_with`, `annotates`.

### Evidence Layer
Every inferred output should include confidence, evidence, and limitations.

Example:
```json
{
  "confidence": 0.82,
  "evidence": [
    "cylindrical face",
    "two circular boundary edges",
    "concave adjacency"
  ],
  "limitations": [
    "STEP does not preserve feature tree",
    "thread/tolerance intent not detected"
  ]
}
```

This keeps the MCP informative instead of over-opinionated.

### Backend Direction
- Keep current `occt-wasm` path for fast baseline metrics during the first refactor.
- Add an advanced graph backend later, preferably Analysis Situs as a sidecar engine.
- Use Analysis Situs for AAG, feature recognition, topology checks, and assembly hierarchy where practical.
- Treat BRepNet, UV-Net, and AAGNet as graph-schema and ML-research references, not initial runtime dependencies.
- Consider NIST SFA, STP2OWL, and OntoBREP ideas for exchange metadata, PMI, validation properties, and semantic provenance.

### Output Style
Structured responses should keep this split:
- `facts`: directly measured values.
- `inferences`: candidates with confidence and evidence.
- `warnings`: health, exchange, or standards risks.
- `limitations`: what STEP cannot prove and what should be checked in drawings/native CAD/PMI.

### Non-Goals
- No direct geometry editing.
- No manufacturing-process generation or toolpaths.
- No authoritative tolerance/inspection claims unless PMI explicitly supports them.
- No pretending STEP preserves feature history, mates, configurations, or full design intent.

## Assumptions
- Public MCP response shape can change because the user explicitly allowed output changes.
- `dist/` is generated and should not be manually edited.
- The first implementation should fix cleanup and redundant traversal before introducing caches.
