# LLM-Friendliness Pass For Tool Definitions

## Goal
Make the five MCP tools maximally reliable for off-the-shelf LLMs: complete every parameter description, fix non-ASCII characters, add one worked example per tool, correct two documentation inaccuracies, and reconcile the feature-type enum with what the system actually emits (which also fixes two real correctness bugs).

## Research Basis (why this matters)
- Tool/parameter descriptions are the dominant factor in tool selection and parameter accuracy (Anthropic: "by far the most important factor"; EMNLP 2025 showed description edits change usage 10x+).
- Parameter descriptions should state ROLE (what it filters/controls), not restate the schema type; give format when non-obvious; for optional fields say "omit if X".
- Enums + min/max + defaults reduce guessing; enum/parameter names aligned to common conventions reduce schema-misalignment errors (PA-Tool, ~80% reduction).
- Descriptions are plaintext; markdown is not guaranteed. Use ASCII only (no en-dash, +/-, degree, squared glyphs). Keep single-string and readable.
- The MCP TS SDK (`@modelcontextprotocol/sdk` 1.29.0) `registerTool` config has no `input_examples`. Most portable mechanism: embed one worked example inside the tool description text.

## Confirmed Issues (verified in code)
1. Missing `describe()` on: `result_mode`, `max_results_policy`, `limit`, `offset`, `sample_entity_limit` (step-tools.ts:39-49); `cluster` fields (step-tools.ts ~118-126); all include enums (face/edge/feature, step-tools.ts ~128-187); all group_by enums (step-tools.ts ~189-205); all sort schemas `by`+`direction` (step-tools.ts ~210-231); `featureTypeSchema` (step-tools.ts:371).
2. Non-ASCII characters in descriptions: step-tools.ts lines 78, 84, 99, 105, 253, 256, 262, 351, 354 (en-dash, +/-, degree, squared); index.ts:106 (en-dash). Replace all with ASCII.
3. Unit bug: step-tools.ts:292 edge `length_min` says "mm^2"; should be "mm".
4. Filter-logic doc inaccuracy: tool descriptions say filters use "AND logic", but multi-value type arrays (`surface_type`, `curve_type`, `feature_type`) are OR within the array. Clarify: filters are AND across fields, OR within a single multi-value array.
5. Feature-type enum disconnected from emitters (CORRECTNESS BUG):
   - Emitted but unqueryable: `through_hole_candidate`, `blind_hole_candidate` (aag-provider.ts:136) map to `unknown` in `mapFeatureType` (features.ts:138-147).
   - Dead enum values (no emitter): `cylindrical_region`, `coplanar_face_group`, `parallel_face_pair`.
6. `filter.through` is non-functional: `parameters.through` is never populated (features.ts extract loop ~118-127), so `through` filter and `group_by: through` always match nothing.
7. Always-empty include values mislead the model: `adjacent_faces`/`edge_ids` (faces.ts:227,231), `adjacent_faces` (edges.ts:225), `source_edges`/`adjacent_faces`/`source_faces` (features.ts:330,334 and never-populated source_faces).
